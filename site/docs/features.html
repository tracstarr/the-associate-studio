Feature Reference

A thorough guide to everything The Associate Studio can do — from terminal sessions and git operations to agent orchestration and the Neural Field dashboard.

Claude CLI Sessions

Sessions are the heart of The Associate Studio. Every Claude conversation runs inside a real Windows ConPTY terminal, backed by the portable-pty crate, so you get the exact same behavior as running claude on the command line. There is no output scraping, no fake shell emulation, and no translation layer between you and the CLI. What the terminal renders is what ConPTY produces, character by character.

You can open as many sessions as your machine will handle. Each one appears as a tab in the main editor area, rendered by xterm.js. The critical design choice here is that switching tabs never kills a session. Inactive terminals are hidden with CSS (display: none) rather than unmounted from the React tree. This means a long-running Claude conversation keeps working in the background while you read diffs in another tab or browse your project files in a third.

Live session detection

The Associate Studio installs lightweight CLI hooks into ~/.claude/theassociate/ that fire when Claude sessions start, stop, or spawn subagents. A file watcher on the backend reads these hook events from an append-only hook-events.jsonl file, tracking byte offsets so it only processes new lines. The result is real-time session awareness without polling the process table.

Active sessions show a green pulsing dot next to their tab title. When a session spawns subagents (for example, during parallel tool use), you see a count badge on the tab. Completed sessions transition to an idle state and eventually show a completion indicator.

Resume and fork

When you open a past session from the session list, it appears as a read-only transcript viewer. From there, you can resume it (Ctrl+R) to pick up where you left off in a live terminal. You can also fork a session, which starts a new Claude conversation using the --fork-session flag to branch from an earlier point in the conversation history.

Plan auto-detection

The watcher monitors terminal output for references to plan files. When Claude creates or updates a plan in ~/.claude/plans/, the studio automatically detects it and links the plan to the originating session tab. You can click through to open the plan in the right panel editor without leaving your terminal.

Question notifications

When Claude asks you a question in a session that is not currently focused, the notification system fires. A bell icon appears with an unread badge, and clicking it jumps directly to the session that needs your attention. This is especially useful when running multiple sessions in parallel — you can let three agents work simultaneously and get notified the moment any of them needs human input.

Multi-Project Workspace

The Associate Studio is designed for people who work across multiple codebases simultaneously. The sidebar contains a project dropdown that lists every project you have added. Switch between them and the entire workspace pivots: different tabs, different git status, different session history. Each project is identified internally by encoding its filesystem path (for example, C:\dev\my-project becomes C--dev-my-project), which keeps per-project data cleanly separated in both local storage and the ~/.claude/projects/ directory.

Per-project tab isolation

Tabs are scoped to the active project. When you switch from Project A to Project B, Project A's tabs disappear from the tab bar and Project B's tabs take their place. Nothing is closed or destroyed — the tabs (and their underlying terminals) are simply associated with a different project ID. Switch back and everything is exactly where you left it.

Adding projects

Click the add button in the project dropdown and a native Windows folder picker dialog opens. Select any directory and it becomes a project in the workspace. You can also run claude --init on a project through the command palette to initialize its CLAUDE.md and configuration files.

Project cycling

For fast switching, use Ctrl+Shift+Right and Ctrl+Shift+Left to cycle through your projects without opening the dropdown. This is the kind of shortcut that pays for itself when you are bouncing between a frontend repo, a backend repo, and an infrastructure repo over the course of a single afternoon.

Git Integration

Git support in The Associate Studio is split across two locations in the UI. The sidebar shows a compact status panel with staged, unstaged, and untracked files. The bottom panel provides a richer view with a full commit log, branch tree visualization, and inline diff viewer. Together they cover the common operations without needing to drop to a terminal for routine work.

Status panel

The sidebar git view groups files by state: staged changes, modified but unstaged files, and untracked files. Each file entry shows its status letter (M, A, D, R, etc.) and can be clicked to open an inline diff. The status refreshes automatically when the file watcher detects changes in the working tree.

Diff viewer

Selecting a file opens a side-by-side diff view as a tab. Both staged and unstaged diffs are supported — you choose which to view from the status panel. The diff renderer highlights additions, deletions, and modifications at the line level.

Git log

The bottom panel's Git tab displays the commit history as a scrollable log with branch indicators. Each entry shows the commit hash, author, date, and message. This is especially useful for understanding what Claude sessions have committed and when.

Branch management

Create, checkout, and delete branches directly from the UI. The branch list shows both local and remote tracking branches. Switching branches updates the git status panel immediately.

Worktree support

Git worktrees let you check out multiple branches of the same repository into separate directories simultaneously. The Associate Studio exposes this through dedicated UI for creating new worktrees, listing existing ones, switching the active worktree, and copying files into a new worktree. This is a power-user feature that works particularly well with multi-agent workflows where different Claude sessions work on different branches.

Remote operations

Fetch, pull, and rebase operations run asynchronously and log their output to the Output panel at the bottom of the window. The output includes timestamps and the full git command that was executed, so you always have an audit trail. All git operations are powered by a combination of libgit2 (via the git2 Rust crate) and direct git CLI invocation for operations that libgit2 does not fully support.

Pull Requests & Issues

The Associate Studio pulls together pull requests and issues from three different sources — GitHub, Linear, and Jira — into a single unified panel. Rather than switching between browser tabs for each service, you can triage and review directly within the IDE.

GitHub pull requests

The PR list shows all pull requests for the current repository with filtering by state (open, closed, merged). Each PR entry displays its title, number, author, and status checks. Clicking a PR opens a detail view that includes the description, review comments, and inline code comments. This is backed by the gh CLI, which handles authentication and API calls.

GitHub issues

Issues from the current repository are listed with filtering by label and assignee. Each issue shows its title, number, labels, and current state. You can open an issue to view its full description and comment thread.

Linear issues

If you have configured a Linear API key in Settings, Linear issues appear alongside GitHub data. Filters are available for status, team, and assignee. Linear's richer status model (backlog, todo, in progress, done, canceled) is fully represented.

Jira issues

Jira integration works through an API token and your Jira instance URL. Issues can be filtered by project, status, and assignee. The integration supports Jira Cloud instances and displays issue keys (like PROJ-123), summaries, and status.

All in one place: The bottom panel provides tabs for PRs, GitHub issues, Linear issues, and Jira issues. You do not need to open a browser to check whether a PR has been reviewed or an issue has been assigned.

File Browser & Editor

The sidebar includes a full file tree browser that mirrors the directory structure of your active project. Directories load their contents lazily — only when you expand a folder does the backend read its entries. This keeps the initial load fast even for large monorepos with thousands of files.

Monaco editor

Clicking a file in the tree opens it in a Monaco editor tab. Monaco is the same editor engine that powers VS Code, so you get syntax highlighting for over 20 languages, bracket matching, minimap navigation, find-and-replace, and multi-cursor editing out of the box. The editor integrates with the session store's dirty-state tracking: unsaved changes are marked with a dot on the tab, and you are warned before closing a modified file.

External change detection

When a Claude session or an external tool modifies a file that you have open, the editor detects the change and reloads the content. This keeps your view current without manual refreshing, which matters a great deal when Claude is actively writing code in one tab while you are reading the same file in another.

Notes System

Notes in The Associate Studio are markdown documents that live alongside your Claude data. They come in two scopes: global notes stored in ~/.claude/notes/ that are available across all projects, and per-project notes stored in ~/.claude/projects/{encoded-path}/notes/ that only appear when the corresponding project is active.

Code snippet capture

The notes system integrates directly with the Monaco editor. When you select a range of code in any open file, an "Add to note" button appears. Clicking it captures the selection as a file reference — storing the file path, line range, and the code content itself. These references are embedded in the note's markdown and rendered as anchored code blocks with their source file attribution.

Organization

The Notes panel in the right sidebar lists all notes for the current scope. Each note has a title, creation date, and preview of its content. Notes can be created, edited in a markdown editor, and deleted. File tabs that have associated note references show an indicator, so you can tell at a glance which files you have annotated.

This feature is designed for the workflow of studying a codebase with Claude: you ask Claude questions, read the files it references, and capture the important parts in notes for later reference. The line-anchored snippets mean your notes stay useful even as the code evolves, because you can always trace back to the original source location.

Teams & Agents

The teams system supports Claude's multi-agent orchestration. When you run Claude with team configurations (stored in ~/.claude/teams/), The Associate Studio visualizes the team structure, agent roles, and their current status in the right panel.

Team viewer

Each team configuration is parsed from its JSON file and displayed with its member agents, their roles, and their designated responsibilities. The viewer updates in real time as the file watcher detects changes to team files.

Inbox messaging

Agents within a team communicate through inbox messages stored in ~/.claude/inboxes/. The Associate Studio displays these messages and lets you send new ones, providing visibility into what agents are telling each other during collaborative tasks.

Task tracking

Tasks from ~/.claude/tasks/ are loaded and displayed with their blocking and blocked-by dependencies. This lets you see at a glance which tasks are waiting on others to complete and where the critical path lies in a multi-agent workflow.

Subagent tracking

When a Claude session spawns subagents (child processes for parallel tool execution), the session store tracks each one with its session ID and role. The tab bar shows a badge with the subagent count, and the session list provides details about what each subagent is working on. This tracking is powered by the same hook event system that detects session starts and stops.

Plans & Context

Plans are markdown files that Claude creates during complex tasks, stored in ~/.claude/plans/. The Associate Studio gives them first-class treatment with a dedicated editor and automatic linking to the sessions that created them.

Plan editor

The right panel has a Plans tab that lists all plan files for the current project. Opening a plan gives you a Monaco-powered markdown editor where you can read Claude's plan, make edits, and save changes. A preview mode renders the markdown so you can read it as formatted text rather than raw markup.

Session-plan linking

When the file watcher detects that a session created or referenced a plan file, it stores a link between the plan's filename and the originating session tab. This means you can navigate from a plan back to the session that produced it, and from a session's output directly to its associated plan.

Context panel

The right panel's Context tab shows the project's CLAUDE.md file and any memory files that Claude uses for persistent context. This is useful for understanding what instructions Claude is operating under for the current project, and for editing those instructions without leaving the IDE. The context panel also includes a documentation browser that auto-detects common documentation directories (docs/, doc/, etc.) in the project root.

Session completion summaries

When a Claude session finishes its work, it may produce a completion summary. These summaries are stored as separate files and can be opened as dedicated tabs. They provide a concise record of what the session accomplished, which is valuable for reviewing multi-session workflows after the fact.

Neural Field Dashboard

The Neural Field is a fullscreen overlay that visualizes your entire workspace as an animated node graph. Open it with Ctrl+Shift+Space and the IDE fades away, replaced by a dark canvas where sessions, teams, and agents are represented as glowing nodes connected by edges.

Physics-based layout

Nodes are positioned using a physics simulation with repulsion forces, spring connections, and damping. Active sessions pulse gently. Teams cluster together. The graph settles into a stable layout within a few seconds and adjusts dynamically as sessions start, stop, or spawn subagents. The rendering is done on an HTML5 canvas element for performance — even with dozens of nodes, the animation stays smooth.

HUD counters

The top-right corner of the overlay displays three counters in a monospaced readout: total sessions, total teams, and total agents. Active counts glow gold; zero counts are dimmed. These update in real time as the underlying data changes.

Hover and click interaction

Hovering over a node displays a readout with the node's name and type. Clicking a session node closes the overlay and navigates directly to that session's tab. Clicking a team node switches to the Teams view in the right panel. The overlay closes on Escape or by pressing Ctrl+Shift+Space again.

The Neural Field exists for situational awareness. When you have half a dozen Claude sessions running across three projects with subagents and team coordination, a flat tab bar stops being useful. The spatial visualization lets you see the entire picture at once and jump to wherever you need to be.

Command Palette

Press Ctrl+P from anywhere in the application to open the command palette. It provides fuzzy-matched access to over 25 commands organized into four categories:

View Toggle sidebar, right panel, bottom panel. Switch sidebar views (Sessions, Git, PRs, Issues, Files). Switch right panel tabs (Context, Teams, Plans, Notes). Switch bottom panel tabs (Git, Output, PRs, Issues). Toggle Neural Field.

The palette filters results as you type, matching against both the command label and its description. This is the fastest way to reach any part of the UI without memorizing every keyboard shortcut. The palette always intercepts Ctrl+P, even when focus is inside a terminal, so it serves as a reliable escape hatch to the rest of the application.

Notifications

The notification system serves two purposes: alerting you when Claude asks a question in a background session, and notifying you when sessions complete.

Question notifications

When a Claude session that is not currently focused outputs a question (detected by the hook event system), a notification is created and associated with that session's tab. The notification store tracks read and unread state per tab. An unread badge appears on the bell icon in the status bar, showing the total count of pending questions across all sessions.

Completion notifications

When a session finishes its work, the system can produce a native Windows notification (if enabled in settings) so you are alerted even when the application is minimized or behind other windows.

Navigation

Clicking a notification in the dropdown jumps directly to the session tab that triggered it and marks it as read. The badge count decrements accordingly. This interaction is designed to feel like a messaging app: you see the count, you click, you land in the right place, and the count clears.

Settings & Integrations

Open settings with Ctrl+, or through the command palette. The settings panel is organized into sections covering appearance, integrations, and behavior.

Appearance

  • Font size — adjustable from 8px to 24px, with keyboard shortcuts (Ctrl+=, Ctrl+-, Ctrl+0 to reset to 13px)
  • Font family — configurable for both the UI and the terminal

GitHub integration

Two authentication methods are supported. The OAuth device flow walks you through GitHub's browser-based authorization: the app displays a code, you enter it on github.com, and the token is stored securely. Alternatively, you can paste a Personal Access Token directly. Both methods store the token in Windows Credential Manager, never in a plain file.

Linear integration

Enter a Linear API key in settings to enable issue tracking. The key is stored in Windows Credential Manager and used for all Linear API requests.

Jira integration

Provide your Jira instance URL, email, and API token. The token is stored in Windows Credential Manager. Once configured, Jira issues appear in the bottom panel alongside GitHub and Linear data.

Session hooks

The settings panel includes controls for managing Claude CLI hooks. Hooks are installed automatically on first launch, but you can remove and reinstall them from settings if needed. The hooks are small shell scripts that write lifecycle events to ~/.claude/theassociate/hook-events.jsonl.

Behavior toggles

  • Startup files — configure which files or scripts to load when a project opens
  • Native notifications — enable or disable Windows toast notifications for session events

Security

The Associate Studio handles API tokens for GitHub, Linear, and Jira. Every one of these secrets is stored in Windows Credential Manager using the keyring crate on the Rust side. No token ever touches settings.json or any other plain-text file on disk.

DPAPI encryption

Windows Credential Manager encrypts stored credentials using the Data Protection API (DPAPI), which ties decryption to the currently logged-in Windows user. This means your tokens are not accessible to other users on the same machine, and they are not recoverable from a disk image without your Windows login credentials.

Path traversal protection

All file operations on the Rust backend validate and sanitize paths before accessing the filesystem. Operations are restricted to the project directory and the ~/.claude/ data directory. Attempts to escape these boundaries via .. segments or other traversal techniques are rejected.

Symlink safety

When deleting team configurations or other data files, the backend checks for symlinks to prevent a malicious symlink from causing deletion of files outside the expected directory. This is a defense-in-depth measure against symlink attacks.

Nested Claude isolation

When The Associate Studio spawns a claude process in a PTY, it strips the CLAUDECODE environment variable from the child process's environment. This prevents Claude from detecting that it is running inside another Claude-related tool, which would otherwise alter its behavior. The environment is carefully constructed to give the spawned CLI a clean context.
